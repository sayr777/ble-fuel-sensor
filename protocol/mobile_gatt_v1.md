# Mobile GATT Profile v1

## Назначение

Этот профиль используется для взаимодействия `Android` приложения с датчиком. Через него приложение читает телеметрию и диагностику, управляет конфигурацией и работает с тарировочной таблицей.

## BLE сервис

- `Service UUID`: `9b0a1000-4d7b-42a1-8f6a-7c3e6b510000`

## Характеристики сервиса

| UUID | Имя | Свойства | Назначение |
| --- | --- | --- | --- |
| `9b0a1001-4d7b-42a1-8f6a-7c3e6b510000` | `Telemetry` | `Read`, `Notify` | Текущая телеметрия |
| `9b0a1002-4d7b-42a1-8f6a-7c3e6b510000` | `Diagnostics` | `Read`, `Notify` | Сырые значения и диагностика |
| `9b0a1003-4d7b-42a1-8f6a-7c3e6b510000` | `Command` | `Write`, `Write Without Response` | Команды приложения |
| `9b0a1004-4d7b-42a1-8f6a-7c3e6b510000` | `Response` | `Read`, `Notify` | Ответы датчика |
| `9b0a1005-4d7b-42a1-8f6a-7c3e6b510000` | `Event` | `Notify` | Асинхронные события |

## Формат PDU для команд и ответов

### Request PDU

| Смещение | Размер, байт | Поле |
| --- | ---: | --- |
| 0 | 1 | `protocol_version = 0x01` |
| 1 | 1 | `message_type = 0x10` |
| 2 | 1 | `opcode` |
| 3 | 1 | `transaction_id` |
| 4 | 2 | `payload_length` |
| 6 | N | `payload` |
| 6 + N | 2 | `crc16` |

### Response PDU

| Смещение | Размер, байт | Поле |
| --- | ---: | --- |
| 0 | 1 | `protocol_version = 0x01` |
| 1 | 1 | `message_type = 0x11` |
| 2 | 1 | `opcode` |
| 3 | 1 | `transaction_id` |
| 4 | 1 | `status` |
| 5 | 2 | `payload_length` |
| 7 | N | `payload` |
| 7 + N | 2 | `crc16` |

## Авторизация

- команды чтения могут выполняться без авторизации;
- команды записи, изменения тарировки и сохранения в память требуют `AUTH`;
- после перезагрузки авторизация должна считаться сброшенной;
- для `v1` допускается `service PIN` или одноразовый сервисный токен.

## Команды приложения

### `HELLO (0x01)`

Назначение:

- прочитать версию прошивки;
- узнать поддерживаемые функции;
- узнать максимальное число точек тарировочной таблицы.

Ответ payload:

- `fw_major`
- `fw_minor`
- `fw_patch`
- `capability_flags`
- `max_cal_points`

### `AUTH (0x02)`

Request payload:

- `auth_type`
- `auth_data`

Response payload:

- `session_flags`

### `READ_TELEMETRY (0x10)`

Возвращает тот же набор прикладных полей, что и телеметрический профиль БНСО.

### `READ_DIAGNOSTICS (0x11)`

Возвращает:

- сырое значение емкостного канала;
- отфильтрованное значение;
- внутренние температуры;
- флаги ошибок;
- текущий режим алгоритма;
- время последней стабильной фиксации уровня.

### `READ_ANALYTICS_SNAPSHOT (0x12)`

Возвращает текущую прикладную сводку:

- расход на транспортную работу;
- расход на технологический цикл;
- расход на холостой ход;
- необъясненный расход;
- число подозрительных событий по топливу.

### `READ_CONFIG (0x20)` и `WRITE_CONFIG (0x21)`

Используются для чтения и изменения:

- `installation_id` (`vehicle_net_id`);
- `sensor_id`;
- `tank_id`;
- периода измерения;
- периода BLE публикации;
- профиля топлива;
- единиц отображения;
- порогов предупреждений.

Для `v1` следует считать, что `installation_id`, `sensor_id` и `tank_id` являются параметрами доверенной привязки установки. Базовый способ их настройки - сервисный `USB`. Изменение через `BLE GATT` допустимо только в специальном сервисном режиме и после усиленной авторизации.

### `READ_CAL_INFO (0x30)`

Возвращает метаданные таблицы:

- `calibration_revision`;
- `system_model_revision`;
- `tank_id`;
- число точек;
- дату последней записи;
- признак валидности;
- контрольную сумму таблицы.

### `READ_CAL_POINT (0x31)`

Request payload:

- `point_index`

Response payload:

- `point_index`
- `raw_code`
- `level_percent_x10`
- `volume_l_x10`

### `WRITE_CAL_POINT (0x32)`

Request payload:

- `point_index`
- `raw_code`
- `level_percent_x10`
- `volume_l_x10`

### `DELETE_CAL_POINT (0x33)`

Request payload:

- `point_index`

### `READ_TEMP_COMP (0x34)` и `WRITE_TEMP_COMP (0x35)`

Используются для чтения и записи:

- коэффициента по температуре топлива;
- коэффициента по температуре воздуха;
- базовой опорной температуры.

### `READ_EVENT_FILTERS (0x36)` и `WRITE_EVENT_FILTERS (0x37)`

Используются для чтения и записи параметров:

- порога распознавания долива;
- порога распознавания слива;
- времени подтверждения долива;
- времени подтверждения слива;
- коэффициента сглаживания стабильного режима;
- коэффициента сглаживания для долива;
- коэффициента сглаживания для слива.

### `READ_SYSTEM_MODEL (0x3B)` и `WRITE_SYSTEM_MODEL (0x3C)`

Используются для чтения и записи:

- списка баков;
- списка потребителей;
- привязки потребителей к бакам;
- профиля машины;
- параметров работы на уклоне;
- правил вычисления достоверности.

### `READ_SYSTEM_PROFILE (0x3D)` и `WRITE_SYSTEM_PROFILE (0x3E)`

Используются для чтения и записи полного профиля топливной системы машины:

- состава баков;
- магистралей подачи;
- магистралей обратки;
- маршрутов потребителей;
- правил перекачки;
- описания клапанов и насосов.

### `VALIDATE_SYSTEM_PROFILE (0x3F)`

Запускает проверку профиля на ошибки:

- некорректные связи между баками;
- отсутствующий источник питания потребителя;
- некорректную обратку;
- конфликтующие маршруты;
- баки без привязанного датчика, если такой датчик обязателен.

Ответ должен возвращать:

- `validation_status`;
- число найденных ошибок;
- код первой ошибки;
- ссылку на проблемную сущность профиля.

### `COMMIT_CALIBRATION (0x38)`

Датчик:

- проверяет целостность таблицы;
- сортирует точки;
- пересчитывает контрольную сумму;
- увеличивает `calibration_revision`.

### `SAVE_TO_NVM (0x39)`

Сохраняет:

- конфигурацию;
- таблицу тарировки;
- температурные коэффициенты;
- параметры фильтрации событий.

### `REBOOT (0x3A)`

Перезапускает датчик после подтвержденной записи.

## Асинхронные события

Через характеристику `Event` датчик может отправлять:

- смену `level_state`;
- изменение `validity_status`;
- окончание записи в память;
- ошибку тарировки;
- потерю валидности измерительного канала;
- вход и выход из сервисного режима;
- локально классифицированное событие расхода.
