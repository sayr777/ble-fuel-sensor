# Protocol v1

## Назначение

Этот документ фиксирует общие правила `v1` для всех профилей взаимодействия датчика:

- кодирование чисел;
- общие перечисления;
- флаги качества;
- типы сообщений и команд;
- стандартные статусы ответов.

## Общие правила

- `protocol_version = 0x01`;
- все многобайтные поля кодируются в `Little Endian`;
- отрицательные температуры кодируются в `int16`;
- если значение недоступно, для знаковых полей используется `0x8000`, для беззнаковых полей `0xFFFF`;
- контроль целостности прикладного пакета выполняется с помощью `CRC16-CCITT`, начальное значение `0xFFFF`, полином `0x1021`.

## Базовый набор телеметрических полей

| Поле | Размер, байт | Формат | Комментарий |

| `sensor_id` | 2 | `u16` | Идентификатор ДУТ |
| `tank_id` | 1 | `u8` | Идентификатор бака |
| `system_model_revision` | 1 | `u8` | Ревизия модели баков и потребителей |
| `sequence` | 1 | `u8` | Счетчик кадров |
| `fuel_level_percent_x10` | 2 | `u16` | Уровень топлива в `0.1 %` |
| `volume_l_x10` | 2 | `u16` | Объем топлива в `0.1 л`, прежнее имя `fuel_volume_l_x10` |
| `fuel_temp_c_x10` | 2 | `i16` | Температура топлива в `0.1 °C` |
| `air_temp_c_x10` | 2 | `i16` | Температура воздуха в `0.1 °C` |

## Минимально обязательный состав каждого телематического пакета

Независимо от конкретного транспортного профиля, каждый телематический пакет должен содержать как минимум:

- `installation_id` или `vehicle_net_id`;
- `sensor_id`;
- `tank_id`;
- `sequence`;
- `volume_l_x10`;
- `level_state`;
- `validity_status`;
- `confidence_percent`;
- `quality_flags`.

Остальные поля могут быть обязательными для конкретного профиля, но перечисленный набор считается нижней границей совместимости `ДУТ`, `БНСО` и мобильных средств сервиса.

## Конфигурация идентификации и привязки

Поля `installation_id`, `sensor_id` и `tank_id` считаются параметрами доверенной привязки датчика к конкретной машине и баку. Они должны:

- настраиваться через сервисный `USB` канал;
- сохраняться в энергонезависимой памяти;
- изменяться по `BLE` только в сервисном режиме после усиленной авторизации;
- участвовать в защите от ошибочного приема пакетов от другой машины.

## Типы сообщений

| Код | Имя | Назначение |
| --- | --- | --- |
| `0x01` | `TELEMETRY` | Короткая телеметрия для БНСО |
| `0x02` | `DIAGNOSTIC` | Расширенная диагностика |
| `0x10` | `APP_REQUEST` | Команда от мобильного приложения |
| `0x11` | `APP_RESPONSE` | Ответ датчика мобильному приложению |
| `0x12` | `APP_EVENT` | Асинхронное событие для приложения |

## Состояние уровня

| Код | Имя | Смысл |
| --- | --- | --- |
| `0x00` | `STABLE` | Уровень стабилен |
| `0x01` | `REFILL` | Обнаружен долив |
| `0x02` | `DRAIN` | Обнаружен слив |

## Статус достоверности

| Код | Имя | Смысл |
| --- | --- | --- |
| `0x00` | `VALID` | Данные достоверны |
| `0x01` | `DEGRADED` | Данные пригодны с ограничениями |
| `0x02` | `INVALID` | Данные нельзя использовать как итоговый уровень |
| `0x03` | `UNKNOWN` | Достоверность пока не определена |

## Флаги качества

| Бит | Маска | Значение |
| --- | --- | --- |
| 0 | `0x0001` | Измерение валидно |
| 1 | `0x0002` | Активна фильтрация всплеска |
| 2 | `0x0004` | Значение удержано антидребезгом |
| 3 | `0x0008` | Ошибка датчика температуры топлива |
| 4 | `0x0010` | Ошибка датчика температуры воздуха |
| 5 | `0x0020` | Ошибка емкостного канала |
| 6 | `0x0040` | Сервисный режим |
| 7 | `0x0080` | Применена температурная коррекция |
| 8 | `0x0100` | Применен профиль долива или слива |
| 9 | `0x0200` | Требуется повторная тарировка |
| 10 | `0x0400` | Сильный уклон |
| 11 | `0x0800` | Несогласованность между баками |

| Код | Имя | Смысл |

## Сущности профиля топливной системы

Для конфигурации многобаковой машины протокол должен поддерживать отдельные сущности:

- `tank_entity`
- `line_entity`
- `consumer_entity`
- `return_route_entity`
- `transfer_entity`
- `actuator_entity`

Каждая сущность должна иметь:

- идентификатор;
- ревизию;
- полезную нагрузку конфигурации;
- признак активности;
- признак валидности.
