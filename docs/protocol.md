# Обзор протокола обмена

## Каналы обмена

В `v1` протокол делится на два независимых профиля:

- `BLE telemetry profile` для взаимодействия `ДУТ -> БНСО`;
- `BLE GATT service profile` для взаимодействия `мобильное приложение <-> ДУТ`.

Сервисный `USB` канал может использовать те же прикладные команды, что и мобильное приложение, но в этом документе основной акцент сделан на BLE-взаимодействии.

## Роли участников

- `ДУТ` измеряет уровень топлива, температуры и вычисляет состояние уровня;
- `БНСО` принимает короткие телематические кадры, сопоставляет их с баками и потребителями и передает данные дальше;
- `мобильное приложение` читает расширенную диагностику, настраивает датчик и работает с тарировочной таблицей;
- сервер строит управленческую аналитику по расходу и событиям.

## Общие правила кодирования

- порядок байт для числовых полей: `Little Endian`;
- температуры передаются в формате `int16`, единица `0.1 °C`;
- уровень передается как `uint16`, единица `0.1 %`;
- объем передается как `uint16`, единица `0.1 л`;
- уклоны передаются в формате `int16`, единица `0.1°`;
- доверенность передается как `uint8`, единица `1 %`;
- счетчик `sequence` увеличивается на `1` для каждого нового телематического кадра;
- поле `calibration_revision` увеличивается после каждого подтвержденного изменения тарировочной таблицы;
- поле `system_model_revision` увеличивается при изменении модели баков или потребителей.

## Профиль для БНСО

Для БНСО используется короткий однонаправленный телематический кадр в `BLE advertising manufacturer data`. Такой формат устойчив к потере отдельных пакетов, не требует соединения и хорошо подходит для фонового опроса.

БНСО должен:

- выполнять пассивное или активное сканирование;
- фильтровать кадры по сигнатуре производителя и `sensor_id`;
- учитывать `tank_id`;
- отбрасывать дубликаты по полю `sequence`;
- считать датчик недоступным при отсутствии новых кадров дольше заданного таймаута.

Подробная спецификация находится в:

- [../protocol/bnso_ble_telemetry_v1.md](../protocol/bnso_ble_telemetry_v1.md)

## Профиль для мобильного приложения

Для мобильного приложения используется `BLE GATT service` с отдельными характеристиками для:

- текущей телеметрии;
- диагностики;
- команд;
- ответов;
- потоковых событий.

Через этот профиль приложение должно уметь:

- читать текущие параметры датчика;
- читать сырые измерения и диагностические флаги;
- изменять конфигурацию;
- читать и редактировать тарировочную таблицу;
- настраивать коэффициенты температуры;
- настраивать пороги и параметры для долива и слива;
- читать топологию баков и потребителей;
- читать и редактировать профиль топливной системы;
- читать параметры уклонов и доверенности;
- сохранять изменения в энергонезависимую память.

Подробная спецификация находится в:

- [../protocol/mobile_gatt_v1.md](../protocol/mobile_gatt_v1.md)

## Общие прикладные данные

В обоих профилях используются одни и те же прикладные сущности:

- `fuel_level_percent_x10`
- `fuel_volume_l_x10`
- `fuel_temp_c_x10`
- `air_temp_c_x10`
- `tank_id`
- `level_state`
- `validity_status`
- `confidence_percent`
- `consumer_mask`
- `tilt_pitch_deg_x10`
- `tilt_roll_deg_x10`
- `quality_flags`
- `fuel_profile`
- `calibration_revision`
- `system_model_revision`

## Смысл `level_state`

- `stable` - уровень стабилен;
- `refill` - распознан долив;
- `drain` - распознан слив.

Это поле обязательно для телематики, потому что позволяет БНСО и мобильному приложению видеть не только абсолютный объем, но и интерпретацию динамики уровня.

## Серверная аналитика

Сервер должен работать не только с временным рядом уровня, но и с событиями:

- `refill_event`;
- `drain_event`;
- `steady_consumption`;
- `multi_tank_imbalance`;
- `low_confidence_measurement`.

## Управленческие категории расхода

На уровне `БНСО` и сервера расход должен классифицироваться по категориям:

- `transport_work`;
- `technology_cycle`;
- `idle`;
- `auxiliary_consumption`;
- `unexplained_consumption`;
- `suspected_fuel_theft`.
