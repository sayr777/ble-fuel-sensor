# Модель топливной системы машины

## Назначение

Для коммерческого транспорта и дорожной техники одной тарировочной таблицы недостаточно. Нужна системная модель, которая описывает:

- несколько баков;
- несколько потребителей топлива;
- режимы `stable/refill/drain`;
- работу на уклонах;
- температурную коррекцию;
- версию и ревизию тарировок;
- отдельный статус достоверности;
- серверную аналитику по событиям, а не только по уровню.

## Базовые сущности

### Бак

Для каждого бака нужно хранить:

- `tank_id`;
- тип бака;
- геометрию бака;
- связанную тарировочную таблицу;
- признак ведущего или вспомогательного бака;
- текущий измеренный объем;
- статус достоверности измерения.

### Датчик

Для каждого ДУТ нужно хранить:

- `sensor_id`;
- привязку к `tank_id`;
- версию прошивки;
- ревизию тарировочной таблицы;
- версию модели топливной системы;
- текущий статус связи и валидности.

### Потребитель топлива

Потребитель - это любой агрегат, который реально расходует топливо:

- основной двигатель;
- автономный отопитель;
- холодильная установка;
- генератор;
- гидростанция;
- отдельный двигатель надстройки.

Для каждого потребителя нужны:

- `consumer_id`;
- тип потребителя;
- связанный бак или допустимый набор баков;
- приоритет;
- признак активности;
- источник данных о включении.

## Модель нескольких баков

Система должна поддерживать:

- один датчик на один бак;
- несколько баков на одну машину;
- несколько баков с разной геометрией;
- баки с разной тарировкой;
- независимые и связанные баки.

Если баки связаны между собой, модель должна допускать:

- переток между баками;
- балансировочную магистраль;
- раздельную подачу и обратку;
- переключаемый маршрут питания.

Чтобы эта модель была пригодна в эксплуатации, для машины нужен отдельный профиль топливной системы:

- [fuel-system-profile.md](fuel-system-profile.md)

## Модель нескольких потребителей

Система должна понимать не только общий остаток топлива, но и кто именно мог быть причиной изменения уровня.

Минимально нужно поддерживать:

- битовую маску активных потребителей;
- карту `consumer_id -> source tank`;
- сценарии, когда один потребитель может брать топливо из нескольких баков;
- сценарии, когда несколько потребителей используют один бак.

## Режимы `stable/refill/drain`

Эти режимы должны определяться отдельно по каждому баку.

Нужно хранить:

- текущий режим;
- время входа в режим;
- подтвержденный объем события;
- флаг достоверности события;
- источник классификации: локальный алгоритм или серверная аналитика.

## Работа с уклонами

Для реальной техники уровень топлива сильно зависит от положения машины. Поэтому модель должна учитывать:

- продольный уклон;
- поперечный уклон;
- допустимую зону доверия при уклоне;
- снижение доверенности результата при сильном наклоне;
- отдельные профили для дорожной техники и машин с высокой вибрацией.

Уклон должен влиять не только на график уровня, но и на статус достоверности и правила детекции событий.

## Температурная коррекция

Температурная модель должна использовать:

- температуру топлива;
- температуру воздуха;
- профиль топлива;
- опорную температуру;
- коэффициенты коррекции.

Прикладной результат должен показывать:

- была ли применена температурная коррекция;
- в каком диапазоне температур она считалась валидной;
- не вышел ли канал за пределы доверенной зоны.

## Версия и ревизия тарировок

Для устойчивой эксплуатации нужно разделять:

- `calibration_revision` - ревизию тарировки конкретного бака;
- `system_model_revision` - ревизию модели топливной системы машины;
- `algorithm_revision` - ревизию алгоритма оценки уровня и событий.

Это нужно, чтобы сервер мог понимать, почему поведение графика изменилось после обновления.

## Отдельный статус достоверности

Достоверность результата не должна прятаться только в `quality_flags`. Нужен отдельный прикладной статус:

- `valid`;
- `degraded`;
- `invalid`;
- `unknown`.

На него должны влиять:

- уклон;
- шум;
- пропуски данных;
- ошибки температурных каналов;
- несогласованность между баками;
- незавершенный режим долива или слива.

## Серверная аналитика по событиям

На сервер должен приходить не только уровень, но и события:

- подтвержденный долив;
- подтвержденный слив;
- штатный расход;
- расход на транспортную работу;
- расход на технологический цикл;
- расход на холостой ход;
- расход на отдельный агрегат;
- потеря достоверности;
- восстановление достоверности;
- изменение топливной конфигурации.

Для каждого события полезно хранить:

- `event_id`;
- `event_type`;
- время начала и завершения;
- затронутые баки;
- активные потребители;
- оценку изменения объема;
- достоверность события;
- комментарий алгоритма.

## Управленческая аналитика

Кроме событий по бакам, сервер должен формировать прикладные агрегаты для бизнеса:

- расход на транспортную работу;
- расход на технологический цикл;
- расход на холостой ход;
- подтвержденные и подозрительные изъятия топлива.

Эта логика подробно описана в:

- [analytics-model.md](analytics-model.md)

## Практический вывод

Для сложной техники `ДУТ` должен быть частью общей топливной модели машины. Только в этом случае можно перейти от "уровень в баке" к управленческой аналитике:

- сколько топлива реально осталось;
- куда оно ушло;
- какой агрегат мог его потребить;
- был ли долив или слив;
- насколько результату можно доверять.
